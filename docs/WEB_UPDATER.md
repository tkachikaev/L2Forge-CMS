# Web Updater

Web Updater появился в KaevCMS 0.32.0; в 0.32.1 добавлены сохраняемые этапы и усиленное восстановление и доступен только владельцу панели:

```text
Настройки → Системная информация → Управление обновлениями
```

В 0.32.0 используется ручной режим: владелец самостоятельно загружает доверенный ZIP-пакет. Автоматическая проверка GitHub Releases пока не подключена.

## Обновление через несколько версий

Пакет является кумулятивным и содержит полное состояние целевого релиза, а не только разницу с одной предыдущей версией. В манифесте указывается диапазон исходных версий:

```json
{
  "schema": 1,
  "package_id": "kaevcms-0.33.0",
  "target_version": "0.33.0",
  "minimum_version": "0.32.0",
  "maximum_version": "0.32.10"
}
```

Один такой пакет может обновить, например, 0.32.0, 0.32.4 или 0.32.10 непосредственно до 0.33.0. Laravel выполняет все ещё не применённые миграции базы в правильном порядке.

Переход 0.31.12 → 0.32.0 выполняется старым PowerShell-патчем, поскольку в 0.31.12 Web Updater ещё отсутствует. Первый практический пакет для Web Updater выпускается для версии после 0.32.0.

## Поддерживаемые схемы установки

Один и тот же пакет работает в двух вариантах:

- стандартная установка: корень приложения и `public/` находятся в одном проекте;
- split-установка: закрытое ядро `kaevcms-core/` и публичная папка хостинга находятся отдельно.

Логические пути пакета начинаются с:

```text
core/    → base_path()
public/  → public_path()
```

Web Updater сам определяет текущую схему и раскладывает файлы в правильные каталоги.

## Что проверяется перед установкой

До изменения сайта выполняются проверки:

- установленное расширение PHP `zip`;
- корректный JSON-манифест схемы 1;
- текущая версия входит в поддерживаемый диапазон;
- целевая версия новее текущей;
- версия PHP и обязательные расширения соответствуют пакету;
- в ZIP нет traversal-путей, абсолютных путей, символических ссылок, дубликатов и незаявленных файлов;
- пакет не пытается заменить `.env`, runtime-хранилище, SQLite-файл, пользовательские загрузки или bootstrap-cache;
- размер и SHA256 каждого payload-файла совпадают с манифестом;
- `core/VERSION` совпадает с `target_version`;
- каталоги назначения доступны для записи;
- база CMS поддерживает автоматическую резервную копию;
- свободного места достаточно для staging и резервных копий.

SHA256 защищает от повреждения пакета, но не доказывает его происхождение: в ручном режиме загружать следует только пакет, полученный из доверенного релиза KaevCMS. Цифровая подпись и автоматический канал обновлений относятся к следующему этапу.

## Процесс установки

После подтверждения текущим паролем владельца Web Updater:

1. заранее показывает владельцу аварийную ссылку обхода режима обслуживания;
2. устанавливает атомарную блокировку, запрещающую параллельный запуск;
3. повторно проверяет пакет, SHA256 всего ZIP и текущее состояние;
4. включает режим обслуживания;
5. создаёт резервную копию базы CMS и сохраняет проверяемый манифест backup-файла;
6. копирует изменяемые и удаляемые файлы в закрытый каталог резервной копии и сохраняет манифест отката;
7. сохраняет этап `files`, атомарно заменяет файлы и удаляет заявленные устаревшие пути;
8. перед миграциями сохраняет этап `migrations` и выполняет `php artisan migrate --force` через Laravel;
9. сохраняет этап `finalizing`, очищает кэши и отправляет `queue:restart`;
10. записывает целевую версию в `VERSION`, marker и `cms_settings`;
11. выключает режим обслуживания и сохраняет результат в журнале.

Внешние базы LoginServer и GameServer в резервную копию Web Updater не входят и не изменяются миграциями ядра.

## Откат

Если установка завершается исключением, Web Updater определяет сохранённый этап и восстанавливает только то, что могло измениться. До первого действия отката он проверяет наличие всех обязательных резервных копий. Если нужной копии нет, частичный откат не запускается. Затем Web Updater пытается:

- восстановить резервную копию базы CMS;
- вернуть изменённые и удалённые файлы;
- восстановить прежнюю установленную версию;
- снять режим обслуживания;
- записать ошибку и результат отката.

Перед запуском страница показывает аварийную ссылку. Если веб-запрос оборвался и сайт остался в режиме обслуживания, владелец открывает эту ссылку в том же браузере, возвращается на страницу обновления и запускает ручное восстановление. Эксклюзивная блокировка не позволит начать восстановление, пока исходный процесс ещё работает.

Автоматический откат снижает риск, но не заменяет внешнюю резервную копию. Перед крупным production-обновлением рекомендуется иметь отдельную копию базы, `.env`, `public/uploads`, установленных тем и модулей.

Резервные копии Web Updater хранятся в:

```text
storage/app/kaevcms/update-backups/<uuid>/
```

Журналы находятся в:

```text
storage/logs/update-<uuid>.log
```

Они не доступны через HTTP при корректной standard или split-установке.

## Формат пакета

Корень ZIP:

```text
kaevcms-update.json
payload/core/...
payload/public/...
```

Каждый файл манифеста содержит:

```json
{
  "source": "payload/core/app/Example.php",
  "target": "core/app/Example.php",
  "sha256": "...64 hex characters...",
  "size": 1234
}
```

`source` обязан точно соответствовать `payload/<target>`. Устаревшие файлы перечисляются полными логическими путями в массиве `delete`. Wildcard-пути не поддерживаются.

## Сборка релизного пакета

На Windows:

```powershell
.\deployment\windows\build-web-update-package.ps1 `
    -TargetReleaseRoot "C:\Releases\KaevCMS-0.33.0" `
    -MinimumVersion "0.32.0" `
    -MaximumVersion "0.32.10" `
    -PreviousReleaseRoot "C:\Releases\KaevCMS-0.32.10"
```

Результат:

```text
dist/KaevCMS-update-to-0.33.0.zip
```

История удаляемых путей хранится по версиям в:

```text
deployment/updates/deletions.json
```

При передаче `-PreviousReleaseRoot` сборщик сравнивает предыдущий и целевой релизы, автоматически добавляет исчезнувшие пути в историю и включает в пакет объединение удалений для всего заявленного диапазона. Пакет не должен объявлять диапазон через релиз, в котором менялись Composer-зависимости: после такого релиза `minimum_version` поднимается до первой версии с тем же `composer.lock`, что и у целевой сборки.

Web Updater 1.0 намеренно не заменяет `vendor/`. Если целевой `composer.lock` отличается от установленного, пакет отклоняется: смена Composer-зависимостей пока выполняется полным развёртыванием. Это исключает запуск миграций новым кодом через уже загруженный старый Composer-autoloader.

Перед публикацией обязательно выполняются:

```powershell
.\deployment\windows\quality.ps1
.\deployment\windows\browser-quality.ps1
```

а также тестовая установка пакета на копии базы и файлов поддерживаемой старой версии.

## Ограничения 0.32.1

- нет автоматического запроса новой версии из интернета;
- нет цифровой подписи релизного манифеста;
- пакет выбирается владельцем вручную;
- резервная копия MySQL создаётся средствами PHP и рассчитана на базу CMS, а не на большие игровые базы;
- обновление должно выполняться одним владельцем; при обрыве запроса используется аварийная ссылка и ручное восстановление;
- изменение Composer-зависимостей требует полного развёртывания;
- будущие релизы обязаны сохранять обратную совместимость таблицы `system_updates` и updater-контракта схемы 1 в пределах поддерживаемой ветки.
