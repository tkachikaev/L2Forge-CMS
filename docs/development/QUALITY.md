# Контроль качества

Перед публикацией изменений выполните:

```powershell
.\quality.ps1
```

Скрипт работает без сети: проверяет PowerShell-сценарий обновления, валидность `composer.json`, политику dependency audit, а затем запускает:

1. `composer lint` — Laravel Pint в режиме проверки;
2. `composer analyse` — Larastan/PHPStan;
3. `composer test` — полный набор PHPUnit/Laravel tests;
4. сборку и очистку route cache.

Для быстрой проверки только PHP-кода по-прежнему можно выполнить `composer quality`.

`quality.ps1` временно устанавливает `COMPOSER_DISABLE_NETWORK=1`, не обращается к Packagist и после завершения восстанавливает исходное состояние PowerShell-сеанса. Проверку актуальной базы уязвимостей запускайте отдельно при доступном интернете:

```powershell
.\security-audit.ps1
```

Этот необязательный для ежедневной разработки скрипт выполняет строгие `composer audit --locked --no-interaction` и `npm audit --audit-level=high`. Его достаточно запускать перед релизом, после обновления зависимостей или периодически на сервере/в CI. Он временно разрешает Composer сетевой доступ, даже если перед этим в том же окне запускался `quality.ps1`. Сетевой сбой, найденные advisories или повреждённый lock-файл завершают отдельный аудит ошибкой. GitHub Actions также сохраняет строгие сетевые audit-проверки.

## Статический анализ

Конфигурация находится в `phpstan.neon`. Начальный уровень — 5, анализируются каталоги:

```text
app
routes
database
```

Новые ошибки не следует скрывать массовым baseline. Если правило нельзя исправить сразу, исключение должно быть точечным и документированным.

## Форматирование

Для автоматического исправления оформления используйте:

```powershell
.\vendor\bin\pint
```

Для проверки без изменения файлов:

```powershell
composer lint
```

## Тесты

Каждая новая функция и исправление регрессии должны сопровождаться тестом. Релиз не считается готовым при падении любой проверки `composer quality`.

Для основных моделей используйте фабрики из `database/factories`. Тесты точного количества серверов и удаления последнего GameServer должны создавать изолированную серверную пару через `Tests\Concerns\InteractsWithServerFixtures`, поскольку историческая миграция добавляет начальную строку для совместимости установок.

Подробные правила fixtures, состояний мониторинга и конкурентных сценариев: `docs/development/TESTING.md`.

## Браузерные smoke-тесты

Функции, зависящие от настоящего JavaScript и `wire:navigate`, дополнительно проверяются в Chromium через Playwright:

Первичная подготовка, которой нужен интернет, выполняется один раз:

```powershell
.\browser-setup.ps1
```

После установки npm-зависимостей и Chromium браузерные тесты запускаются без сети:

```powershell
.\browser-quality.ps1
```

Скрипт подготавливает отдельную временную SQLite-базу, запускает CMS в окружении `testing` на свободном локальном порту и удаляет тестовую базу после завершения. Рабочая база и `.env` не изменяются. Для редкого Windows-сбоя `ERR_NO_BUFFER_SPACE` повторяется только неудачная локальная навигация, максимум два раза; остальные ошибки не маскируются.

GitHub Actions выполняет оба dependency audit отдельно: Composer по `composer.lock`, npm по `package-lock.json` без загрузки браузера.
