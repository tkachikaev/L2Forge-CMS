# Журнал действий

Журнал действий предназначен для владельцев и администраторов L2Forge CMS. Он показывает важные изменения в панели управления, события пользователей сайта, отправку почты и системные операции.

Адрес раздела:

```text
/admin/logs
```

## Что записывается

В базовой версии журнал фиксирует:

- успешные и неудачные входы существующих администраторов;
- выходы из панели управления;
- создание, изменение, включение и отключение администраторов, а также смену их паролей;
- создание, изменение и удаление новостей;
- изменение основных настроек сайта;
- добавление, изменение и удаление игровых серверов;
- изменение настроек регистрации и SMTP;
- успешную и неуспешную отправку тестового письма;
- изменение, сброс и тестовую отправку шаблонов писем;
- отправку уведомления об успешной смене пароля;
- активацию темы;
- регистрацию, вход и выход пользователей сайта;
- подтверждение email и смену пароля;
- включение и отключение пользователей администратором;
- повторную отправку подтверждения email и ссылки восстановления пароля из карточки пользователя;
- отправку писем подтверждения и восстановления доступа;
- очистку устаревших записей журнала.

По мере развития CMS и модулей список событий будет расширяться.

## Категории

В верхней части раздела доступны категории:

```text
Все
Администраторы
Пользователи
Почта
Система
```

Модули могут добавлять собственные категории. Они автоматически появляются в списке после первой записи.

## Подробности записи

Для каждой записи показываются:

- дата и точное время;
- инициатор действия;
- человекочитаемое название и технический код события;
- объект действия;
- результат: `Успешно` или `Ошибка`;
- IP-адрес;
- User-Agent браузера;
- дополнительные сведения и изменённые значения.

Записи доступны только авторизованным администраторам. Изменять или удалять отдельные записи через панель нельзя.

## Какие данные не сохраняются

Журнал автоматически скрывает чувствительные поля. В него не должны попадать:

- пароли пользователей и администраторов;
- SMTP-пароль;
- пароли игровых и CMS-баз данных;
- `APP_KEY`;
- токены подтверждения email;
- токены восстановления пароля;
- cookies и заголовки авторизации;
- приватные и API-ключи;
- полное содержимое писем.

Для произвольной HTML-отправки журнал сохраняет только получателя, тему, размер HTML, результат и класс исключения при ошибке. Сам HTML-код письма в журнал не записывается.

Вместо значения секретного поля сохраняется отметка:

```text
[REDACTED]
```

## Защита журнала входов

Форма входа администратора использует два независимых уровня ограничения:

```text
IP: 10 запросов в минуту и 100 запросов в час
email + IP: 5 проверок пароля за 60 секунд
```

Значения можно изменить в `.env`:

```dotenv
ADMIN_LOGIN_MAX_ATTEMPTS=5
ADMIN_LOGIN_DECAY_SECONDS=60
ADMIN_LOGIN_IP_MAX_ATTEMPTS_PER_MINUTE=10
ADMIN_LOGIN_IP_MAX_ATTEMPTS_PER_HOUR=100
```

IP-лимит применяется до контроллера. Отклонённые им запросы не создают строки в `admin_login_logs` и `audit_logs`. Попытки с неизвестным email сохраняются только в `admin_login_logs`; общий журнал получает события только для существующей учётной записи администратора.

Если сайт работает за reverse proxy или CDN, веб-сервер должен передавать реальный адрес клиента, а доверенные прокси должны быть настроены корректно. Иначе все посетители могут определяться одним IP.

## Срок хранения

По умолчанию общий журнал хранится 90 дней, а журнал входов администраторов — 30 дней:

```dotenv
AUDIT_LOG_RETENTION_DAYS=90
ADMIN_LOGIN_LOG_RETENTION_DAYS=30
```

Проверить количество устаревших записей без удаления:

```powershell
php artisan l2forge:logs-clean --dry-run
```

Удалить записи старше установленных сроков:

```powershell
php artisan l2forge:logs-clean
```

Можно временно указать другие сроки:

```powershell
php artisan l2forge:logs-clean --days=180 --admin-login-days=60 --dry-run
php artisan l2forge:logs-clean --days=180 --admin-login-days=60
```

Удаление выполняется порциями по 1000 строк, чтобы не создавать длительную блокировку большой таблицы. После реальной очистки в журнал добавляется системная запись с количеством удалённых строк из обеих таблиц.

Команда также зарегистрирована в Laravel Scheduler и запускается ежедневно в `03:30`. На production-сервере планировщик Laravel должен вызываться системным cron или планировщиком задач:

```text
php artisan schedule:run
```

На локальном компьютере отсутствие системного расписания не мешает работе CMS — очистку можно запускать вручную.

## Отличие от laravel.log

```text
Журнал действий
└─ понятные события администраторов, пользователей и CMS

storage/logs/laravel-YYYY-MM-DD.log
└─ технические сообщения Laravel, исключения и отладочная информация
```

По умолчанию технические логи ротируются ежедневно и хранятся 14 дней:

```dotenv
LOG_CHANNEL=stack
LOG_STACK=daily
LOG_DAILY_DAYS=14
```

При расследовании ошибки сначала смотрят журнал действий, а затем при необходимости технический файл из `storage/logs`.
