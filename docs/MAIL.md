# Почтовая система

Почтовые настройки доступны администратору:

```text
/admin/settings/mail
```

Раздел состоит из внутренних вкладок:

```text
Подключение
Доставка
Подтверждение email
Восстановление пароля
Пароль изменён
Отправить письмо
```

## Подключение SMTP

Во вкладке «Подключение» настраиваются:

```text
SMTP-сервер
Порт
STARTTLS / TLS, SSL / SMTPS или соединение без принудительного шифрования
Имя пользователя SMTP
Пароль SMTP
Email отправителя
Имя отправителя
Email для системных уведомлений
```

Имя пользователя и пароль можно оставить пустыми, если почтовый сервер не требует авторизацию.

После сохранения необходимо отправить тестовое письмо. Любое изменение параметров SMTP сбрасывает результат предыдущей проверки. До новой успешной проверки нельзя включить регистрацию с обязательным подтверждением email.

## Шаблоны писем

KaevCMS содержит готовые стандартные шаблоны. После установки ничего настраивать вручную не требуется.

Администратор может изменить:

```text
Тему письма
Название в тёмной шапке письма
Заголовок
Основной текст
Текст кнопки
Дополнительный текст
```

HTML-каркас письма, адаптивная вёрстка и реальная ссылка кнопки формируются CMS автоматически. HTML-теги в полях шаблона запрещены. Для простого выделения допускается Markdown:

```text
**жирный текст**
*курсив*
```

Новый абзац создаётся пустой строкой.

Поле «Название в шапке письма» управляет надписью в тёмной верхней части стандартного оформления. Заводское значение `{{site_name}}` можно заменить любым названием проекта отдельно для каждого шаблона и языка.

## Произвольное HTML-письмо

Адрес раздела:

```text
/admin/settings/mail/custom
```

Форма отправляет одно письмо одному получателю и содержит поля получателя, темы и полного HTML-кода. В редактор заранее подставлен рабочий пример с тёмной шапкой, таблицей и кнопкой. Предпросмотр выполняется в изолированном `iframe`.

Разрешены обычные конструкции почтовой вёрстки: полный документ `html/head/body`, таблицы, изображения, ссылки, блоки `style` и inline CSS. Перед отправкой блокируются PHP, Blade, `script`, `iframe`, формы, JavaScript-обработчики, опасные схемы URL и CSS-выражения.

Произвольное письмо не сохраняется как системный шаблон и не используется для массовой рассылки. SMTP должен быть предварительно настроен и успешно проверен.

## Доступные переменные

Переменные записываются в двойных фигурных скобках:

```text
{{site_name}}
{{site_url}}
{{username}}
{{user_email}}
{{support_email}}
```

Для подтверждения email также доступны:

```text
{{verification_url}}
{{expires_in}}
```

Для восстановления пароля:

```text
{{reset_url}}
{{expires_in}}
```

Неизвестные переменные не сохраняются. Реальные токены и ссылки создаются самой CMS и не показываются администратору.

## Стандартные значения

Стандартные шаблоны хранятся в коде:

```text
config/mail_templates.php
```

Пользовательские изменения сохраняются в таблице `cms_settings`. Обновление CMS не перезаписывает изменённые тексты.

Кнопка «Восстановить стандартный шаблон» удаляет пользовательское переопределение и возвращает заводской вариант из `config/mail_templates.php`.

## Предпросмотр и тестовая отправка

Каждая вкладка шаблона содержит:

```text
Предпросмотр
Отправить тестовый шаблон
```

Предпросмотр использует демонстрационные значения. Тестовая отправка создаёт безопасную примерную ссылку и не выпускает настоящий токен подтверждения или восстановления.

Перед тестовой отправкой шаблона SMTP должен быть настроен и проверен во вкладке «Подключение».

## Уведомление после смены пароля

После успешной установки нового пароля CMS отправляет отдельное уведомление. Ошибка этого уведомления не отменяет уже выполненную смену пароля, но фиксируется в журнале.

## Хранение SMTP-пароля

Пароль SMTP:

- шифруется средствами Laravel с использованием `APP_KEY`;
- не выводится обратно в HTML;
- не записывается в журнал;
- не изменяется, если при повторном сохранении поле пароля оставлено пустым.

Если `APP_KEY` потерян или заменён, старый SMTP-пароль расшифровать нельзя. Потребуется ввести его заново.

## Журналирование

Журнал фиксирует:

```text
mail.template_updated
mail.template_reset
mail.template_test_sent
mail.template_test_failed
mail.custom_sent
mail.custom_failed
settings.mail_delivery_mode_updated
mail.background_probe_started
mail.background_probe_failed
mail.delivery_probe_started
mail.delivery_probe_failed
mail.password_changed_sent
mail.password_changed_failed
```

Полное содержимое шаблонов, реальные ссылки и токены в журнал не записываются. Для изменения шаблона сохраняются только его название и перечень изменённых полей.

## DNS и адрес сайта

Для реального домена рекомендуется настроить SPF, DKIM и DMARC у почтового провайдера.

`APP_URL` должен содержать настоящий публичный адрес сайта. Он используется при формировании подписанных ссылок подтверждения email и восстановления пароля.
Во вкладке «Доставка» KaevCMS предупреждает, если `APP_URL` отличается от адреса, открытого в браузере. Для фоновых заданий URL формируется без HTTP-запроса, поэтому неверный `APP_URL` может направить пользователя на другую установку CMS, где токен отсутствует.

Для восстановления пароля действителен только последний созданный токен. Если пользователь запросил новую ссылку, ещё не отправленное старое письмо автоматически пропускается очередью. Срок действия токена остаётся 60 минут.

## Режим отправки

Во вкладке «Доставка» доступны три режима автоматических писем кабинета:

```text
Синхронный
Асинхронный
Асинхронный с очередью в базе данных
```

Синхронный режим используется при чистой установке. Пользователь ждёт, пока SMTP примет письмо, зато дополнительная настройка сервера не требуется.

Асинхронный режим использует встроенное подключение Laravel `background`: HTTP-ответ возвращается пользователю сразу, а письмо отправляется отдельным PHP-процессом после ответа. Постоянный `queue:work`, Supervisor и Redis для этого режима не требуются. Задание не сохраняется в очереди и при аварийном завершении процесса может остаться необработанным; такая отправка будет видна в мониторинге как ожидающая.

Асинхронный режим с очередью в базе данных использует стандартное Laravel-подключение `database`. Задание сохраняется в таблице `jobs`, переживает перезапуск приложения и повторяется после временной ошибки. Для обработки должен работать Laravel Scheduler либо отдельный worker:

```text
php artisan queue:work database --queue=mail-probe,mail,default
```

KaevCMS ежеминутно запускает команду `kaevcms:queue-drain`. Она обнаруживает все очереди, в которых реально есть задания, и поочерёдно передаёт каждую из них отдельному короткоживущему `queue:work ... --stop-when-empty`. Так одна большая очередь не блокирует остальные. Поэтому при уже настроенном системном `schedule:run` отдельный постоянный worker не обязателен, а будущие очереди модулей не требуют изменения системного расписания.

При первом выборе асинхронного режима CMS автоматически запускает реальную тестовую задачу без отправки письма. Обычный асинхронный тест ожидается до 15 секунд, тест очереди в базе данных — до 90 секунд. Режим включается только после успешного выполнения. При неудаче синхронная отправка остаётся активной.

Выбранный режим применяется к:

```text
подтверждению email
восстановлению пароля
уведомлению об изменении пароля
```

Тест SMTP, тесты шаблонов и произвольные письма администратора намеренно остаются синхронными, чтобы результат операции был известен сразу.

Значение `QUEUE_CONNECTION` в `.env` менять не требуется: KaevCMS явно выбирает `background` или `database` для автоматических писем.


## Настройка асинхронной очереди на Linux VDS

Для режима «Асинхронный с очередью в базе данных» на VDS рекомендуется постоянный Laravel worker под Supervisor. Простая загрузка файлов CMS на сервер сама по себе не запускает `queue:work`: Supervisor или системный cron нужно настроить один раз на стороне хостинга. Примеры ниже рассчитаны на Ubuntu/Debian. Замените путь `/var/www/kaevcms` и пользователя `www-data` на значения своего сервера.

Установите Supervisor:

```bash
sudo apt update
sudo apt install supervisor
```

Создайте файл `/etc/supervisor/conf.d/kaevcms-mail.conf`:

```ini
[program:kaevcms-mail]
process_name=%(program_name)s_%(process_num)02d
directory=/var/www/kaevcms
command=/usr/bin/php /var/www/kaevcms/artisan queue:work database --queue=mail-probe,mail,default --sleep=3 --tries=3 --backoff=10 --max-time=3600
autostart=true
autorestart=true
stopasgroup=true
killasgroup=true
user=www-data
numprocs=1
redirect_stderr=true
stdout_logfile=/var/www/kaevcms/storage/logs/queue-worker.log
stopwaitsecs=60
```

Примените конфигурацию:

```bash
sudo supervisorctl reread
sudo supervisorctl update
sudo supervisorctl start kaevcms-mail:*
sudo supervisorctl status
```

После обновления KaevCMS перезапускайте worker, чтобы он загрузил новый код:

```bash
cd /var/www/kaevcms
php artisan queue:restart
```

Альтернативный упрощённый вариант — Laravel Scheduler через Cron. KaevCMS уже регистрирует одноразовую обработку почтовой очереди каждую минуту:

```cron
* * * * * cd /var/www/kaevcms && /usr/bin/php artisan schedule:run >> /dev/null 2>&1
```

Cron проще и подходит небольшому проекту, но постоянный Supervisor worker быстрее забирает письма и лучше подходит для публичного сервера. Параметр `--max-time=3600` завершает worker примерно раз в час; Supervisor сразу запускает его снова, благодаря чему процесс регулярно подхватывает новый код и освобождает накопленную память. На обычном shared-хостинге, где Supervisor недоступен, используйте cron из панели управления. Если хостинг не разрешает ни cron, ни фоновые процессы, выбирайте синхронный или встроенный `background`-режим. После настройки вернитесь во вкладку «Доставка» и запустите проверку режима с очередью в базе данных. CMS включит его только после фактической обработки тестового задания.

## Контроль доставки

CMS сохраняет только техническое состояние автоматического письма: получателя, тип, режим, время и класс ошибки. Текст письма, токены и ссылки в таблицу мониторинга не записываются.

Дашборд администратора показывает:

```text
текущий режим
количество ожидающих писем
ошибки за последние семь дней
самое старое ожидающее письмо
последнюю успешную отправку
```

Если письмо находится в ожидании больше двух минут, выводится предупреждение. Это не запускает повторную отправку автоматически и не создаёт риск двойного письма.

## Локальная разработка

До настройки SMTP применяется безопасный почтовый транспорт `log`. Регистрация с обязательным подтверждением email недоступна, пока администратор не сохранит SMTP и не выполнит успешную тестовую отправку.

## Язык письма

Шаблоны подтверждения email, восстановления и уведомления о смене пароля имеют отдельную версию для каждого включённого языка. Русский и английский заводские варианты готовы сразу после установки.

Язык письма берётся из `users.locale`. Подписанные ссылки и ссылки восстановления ведут на локализованные маршруты. Для дополнительного языкового пакета администратор может заполнить свой вариант шаблона; пока он не заполнен, применяется резервный язык.

Ключи пользовательских шаблонов имеют вид:

```text
mail.template.email_verification.en.subject
mail.template.password_reset.ru.body
```


## Диагностика и хранение данных очереди

Страница `/admin/settings/system/queue` показывает все активные database-очереди, последнее начало, успешное завершение и окончательную ошибку обработки. Полный payload задания и текст исключения в интерфейсе не выводятся.

Администратор может повторить выбранное неудачное задание, удалить его запись, отправить стандартный сигнал `queue:restart` и вручную запустить очистку устаревших служебных данных. Все действия записываются в аудит.

Scheduler ежедневно в 03:45 запускает `kaevcms:queue-clean`. По умолчанию хранятся:

```text
история завершённой доставки почты — 30 дней
failed_jobs — 90 дней
неактивные heartbeat отдельных очередей — 30 дней
```

Ожидающие записи `mail_deliveries` автоматическая очистка не удаляет. После изменения SMTP ранее активный постоянный worker отмечается как требующий перезапуска; свежий одноразовый worker Scheduler снимает предупреждение после успешного задания.
