# Системная информация

Раздел доступен администраторам по адресу:

```text
/admin/settings/system
```

Она предназначена для диагностики установки и подготовки безопасного отчёта для технической поддержки.

## Версия KaevCMS

Единственным источником версии установленной CMS является файл `VERSION` в корне проекта:

```text
X.Y.Z
```

Файл содержит только номер версии без префикса `v`, названия продукта и дополнительных параметров. При установке официального патча файл заменяется вместе с кодом, поэтому раздел «Системная информация» и подвал админки показывают новую версию автоматически.

Версия не хранится в `.env`, базе данных или `composer.json`.

## Настройка мониторинга серверов

Настройка перенесена на страницу `/admin/settings/admin-panel` во вкладку **«Панель администратора»**. Страница системной информации остаётся диагностической и не содержит изменяемых параметров.

На странице панели администратора находится компактная настройка «Интервал обновления статуса». Она определяет, как часто CMS может повторно проверять подключения к базам, порты LoginServer/GameServer и количество игроков онлайн.

Доступны фиксированные значения:

```text
30 секунд
1 минута — по умолчанию
2 минуты
5 минут
```

Минимальное значение ограничено 30 секундами. Произвольный ввод не используется, чтобы случайная настройка не создала лишнюю нагрузку на игровые базы и сетевые службы. Между проверками публичный сайт и дашборд читают сохранённый снимок. Статус «Настроено» означает успешное подключение к совместимой схеме БД. Статус «Онлайн» требует дополнительно доступной сетевой службы. Публичный игровой мир считается доступным только когда эти две проверки успешны и у GameServer, и у связанного LoginServer. Ручной режим обслуживания имеет приоритет над публичным статусом, но не останавливает сами проверки. Сообщение обслуживания хранится отдельно для каждого включённого языка; время окончания при необходимости указывается прямо в тексте. Глобальная настройка публичного онлайна находится над карточками игровых серверов. При отключении блок количества игроков полностью отсутствует в публичном интерфейсе и JSON-ответе, но данные сохраняются в административной панели.

Настройка хранится в `cms_settings`:

```text
server_monitor.refresh_interval_seconds
```

`SERVER_MONITOR_REFRESH_INTERVAL_SECONDS` из `.env` используется как значение по умолчанию, пока настройка не сохранена через панель. Защитная атомарная блокировка остаётся внутренней: её минимум равен 150 секундам, значение по умолчанию — 300 секунд.

Срок актуальности снимка автоматически увеличивается вместе с выбранным интервалом. Поэтому при выборе 5 минут зелёный статус не станет серым раньше следующей допустимой проверки.


## Публичная игровая статистика

Раздел `/statistics` читает игровые данные только через выбранный GameServer-драйвер. Для L2J Mobius CT0 Interlude поддерживаются рейтинги по уровню, PvP, PK и игровому времени, а также текущие герои и владельцы замков. Публикация по умолчанию выключена и настраивается отдельно на вкладке «Статистика» каждого игрового мира. Для уровня, PvP, PK и игрового времени используются независимые лимиты от 1 до 100; герои и владельцы замков выводятся полностью.

Успешный результат кешируется на пять минут. Если игровая база временно недоступна, CMS на 30 секунд прекращает повторные обращения для той же выборки. Это не влияет на мониторинг серверов и не создаёт копию персонажей в базе CMS.

Структура драйвера, используемые поля и ограничения приватности приведены в `docs/GAME_STATISTICS.md`.

## Какие сведения определяются автоматически

При каждом открытии страницы CMS заново получает:

- версию PHP и Laravel;
- версию Composer, когда команда Composer доступна веб-процессу;
- семейство операционной системы, архитектуру и PHP SAPI;
- фактически используемый тип хеша паролей (`Argon2id`, `Argon2i` или `bcrypt`);
- среду Laravel и состояние режима отладки;
- драйверы кэша, сессий, очередей, почты и логов;
- часовые пояса PHP и CMS;
- драйвер, версию и состояние подключения к базе CMS;
- относительный путь, размер, фактический `busy_timeout`, режим журнала и синхронизации SQLite;
- состояние настройки доверенных reverse proxy без вывода их адресов в отчёт поддержки;
- доступность служебных каталогов для реальной записи;
- состояние почтовой конфигурации;
- фактический heartbeat Laravel Scheduler и обработчика очереди;
- количество ожидающих заданий и `failed_jobs`;
- наличие обязательных и дополнительных расширений PHP.

Кнопка «Обновить сведения» повторяет проверки. Постоянный фоновый опрос сервера не используется.

## Диагностика Scheduler и очередей

Команда `kaevcms:scheduler-heartbeat` зарегистрирована в Laravel Scheduler каждую минуту. Она обновляет таблицу `system_heartbeats`, поэтому CMS отличает реально работающий системный cron или Task Scheduler от одной лишь записи задачи в коде.

Перед началом обработки queue job приложение записывает отдельный heartbeat очереди. На дашборде и странице системной информации отображаются:

- время последнего запуска Scheduler;
- последняя реальная активность очереди;
- общее количество записей в `jobs`;
- количество записей в `failed_jobs`;
- возраст самого старого ожидающего задания.

Для синхронной и обычной фоновой отправки отдельный worker не требуется. При использовании очереди в базе данных CMS предупреждает, если любое задание ожидает более двух минут и обработка не подтверждается. Почтовые показатели и почтовые предупреждения остаются в отдельной карточке «Доставка почты», чтобы административный дашборд не дублировал информацию. Отсутствие heartbeat Scheduler более трёх минут также отображается как проблема.

Технический блок доступен только владельцу и администратору с правом просмотра системной информации. Редактор его не видит.

## Диагностика базы CMS и прокси

Карточка базы показывает фактически используемый драйвер и результат настоящего запроса `select 1`. Для SQLite дополнительно читаются реальные значения `PRAGMA busy_timeout`, `PRAGMA journal_mode` и `PRAGMA synchronous`. Если `APP_ENV=production` вместе с SQLite, страница показывает предупреждение перейти на MySQL или MariaDB.

Строка «Доверенные прокси» проверяет `TRUSTED_PROXIES` из `.env`:

- пустое значение считается нормальным для прямого подключения;
- корректные IP/CIDR отображаются как настроенные, но сами адреса не попадают в безопасный отчёт;
- неверные элементы игнорируются и дают предупреждение;
- `*` помечается предупреждением, поскольку доверяет пересылаемым заголовкам от любого прокси.

## Тип хеша паролей

В карточке «Программное обеспечение» отображается фактически выбранный Laravel алгоритм:

```text
Тип хеша: Argon2id
```

Если режим `auto` выбрал bcrypt, потому что активная сборка PHP не поддерживает Argon2id, под значением выводится короткое пояснение:

```text
Argon2id не поддерживается системой.
```

Пояснение не показывается, когда bcrypt выбран явно на системе, где Argon2id доступен.

## Проверка каталогов

Доступность каталогов определяется не только системным флагом `writable`. CMS создаёт небольшой временный файл и проверяет, что запись действительно выполнена. Удаление диагностического файла выполняется отдельно и с повторной попыткой, поэтому кратковременная блокировка файла антивирусом Windows не превращается в ложную ошибку прав записи.

Проверяются:

```text
storage/
bootstrap/cache/
public/uploads/news/
public/uploads/pages/
public/uploads/settings/
```

## Версия Composer

Composer обычно не является PHP-зависимостью приложения. CMS пытается безопасно выполнить фиксированную команду `composer --version`. Если веб-процесс не видит Composer в `PATH` или запуск системных команд запрещён, выводится:

```text
Не удалось определить
```

Это не считается ошибкой установки. Наличие Composer отдельно проверяет `doctor.ps1`.

## Безопасный отчёт

Кнопка «Скопировать отчёт» формирует текст с версиями и общими параметрами окружения. В отчёт намеренно не включаются:

- `APP_KEY`;
- пароли и токены;
- cookies и заголовки авторизации;
- SMTP-пароль;
- логины и пароли баз данных;
- полные строки подключения;
- абсолютные пути пользователя и сервера;
- содержимое `.env`.

Такой отчёт можно приложить к обращению о проблеме с установкой или обновлением.

## Скрипты установки и диагностики

`setup.ps1`, `update.ps1` и `doctor.ps1` читают номер версии релиза из файла `VERSION`. Успешно установленная версия отдельно записывается в `storage/app/kaevcms/installed-version.json` и в `cms_settings`, поэтому патч не доверяет уже перезаписанному файлу `VERSION`. Установщик и обновление проверяют поддержку выбранного `HASH_DRIVER`. Режим `auto` предпочитает Argon2id и использует bcrypt только при его отсутствии; явно выбранный неподдерживаемый Argon2/Argon2id не заменяется молча и останавливает запуск с диагностикой. Если файл отсутствует или содержит некорректное значение, установка или диагностика сообщит об ошибке сборки.

При обновлении сначала проверяется исходная версия, затем PHP-файлы старого Laravel-кэша удаляются без загрузки приложения. CMS входит в maintenance mode до изменения зависимостей, Composer устанавливает зафиксированные версии с `--no-scripts`, после чего отдельно перестраивается autoload и выполняется package discovery. Пользовательский `.env` updater не изменяет. Текущее состояние maintenance mode определяется через Laravel, поэтому уже включённый администратором file/cache-режим сохраняется. Updater снимает режим в `finally` только тогда, когда включил его самостоятельно. Устаревшие apply-скрипты и legacy-файлы перед PHPUnit переносятся из активного дерева в `storage/app/kaevcms/update-backups`, а при незавершённом обновлении pending-маркер позволяет безопасно повторить тот же apply-скрипт. PHPUnit использует отдельный cache/array maintenance store, поэтому реальный режим обслуживания сайта не превращает все HTTP-проверки в ответы `503`.

## Проверка языков

`doctor.ps1` проверяет наличие встроенных пакетов `ru` и `en`, корректность их JSON и одинаковый набор системных ключей. Дополнительный пакет может быть неполным: отсутствующие строки будут выводиться на резервном языке.

## Контекстная проверка окружения

`doctor.ps1` читает фактические значения `.env` и проверяет наличие heartbeat-задачи и обработчика почтовой очереди в `php artisan schedule:list`. Для SQLite он проверяет `pdo_sqlite` и настроенный файл базы, для MySQL/MariaDB — `pdo_mysql`. При `GAME_ADAPTER=mobius` `pdo_mysql` проверяется дополнительно. SQLite-файл не считается обязательным при `DB_CONNECTION=mysql`.

Потенциально опасные production-настройки выводятся как `[WARN]`, но не превращают диагностический запуск в ложное падение. Проверяются `APP_ENV`, `APP_DEBUG`, `APP_URL`, `APP_FORCE_HTTPS`, `SESSION_SECURE_COOKIE` и `LOG_LEVEL`.

## Системный запуск Scheduler

Статусы LoginServer/GameServer и онлайн игроков обновляются самой CMS при посещении публичной главной или административного дашборда. Для этой функции отдельный cron не обязателен.

Laravel Scheduler нужен для автоматической очистки журналов, обработки выбранной администратором почтовой очереди в базе данных и для постоянных проверок серверов в периоды, когда сайт никто не открывает. Примеры для Linux cron и Windows Task Scheduler приведены в `docs/PRODUCTION.md`.
