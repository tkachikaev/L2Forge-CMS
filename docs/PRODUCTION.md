# Production-развёртывание

Этот раздел описывает минимальную конфигурацию одного экземпляра KaevCMS. Горизонтальное масштабирование и отказоустойчивый кластер требуют отдельной инфраструктуры.

## Обязательные настройки `.env`

Для публичного сайта проверьте как минимум:

```env
APP_ENV=production
APP_DEBUG=false
APP_URL=https://example.com
APP_FORCE_HTTPS=true
SESSION_SECURE_COOKIE=true
LOG_LEVEL=warning
```

После изменения `.env` очистите кэш конфигурации:

```powershell
php artisan optimize:clear
```

`doctor.ps1` выводит предупреждения, если production-конфигурация оставлена с отладкой, HTTP, небезопасными cookie или уровнем логирования `debug`.


## Cloudflare и reverse proxy

Если запросы приходят напрямую на nginx/Apache, оставьте список пустым:

```env
TRUSTED_PROXIES=
```

Если перед сайтом находится Cloudflare, балансировщик или другой reverse proxy, перечислите доверенные IP-адреса и CIDR-диапазоны через запятую:

```env
TRUSTED_PROXIES=127.0.0.1,10.0.0.0/8
```

KaevCMS принимает `X-Forwarded-For`, `X-Forwarded-Host`, `X-Forwarded-Port` и `X-Forwarded-Proto` только от этих адресов. Это позволяет журналу и ограничениям входа видеть реальный IP посетителя и корректно определять HTTPS. Значение `*` доверяет любому прокси и допустимо только когда веб-сервер невозможно открыть напрямую из интернета. После изменения выполните `php artisan optimize:clear`.

## Хеширование паролей

По умолчанию используется режим наилучшего доступного алгоритма:

```env
HASH_DRIVER=auto
HASH_VERIFY=true
BCRYPT_ROUNDS=12
```

`auto` выбирает Argon2id, когда текущий `php.exe` показывает `argon2id` в `password_algos()`. Если конкретная PHP-сборка не содержит Argon2id, используется bcrypt с cost 12. Это свойство PHP-сборки, а не Composer-зависимость. Для диагностики выполните:

```powershell
php -r "print_r(password_algos());"
php --ini
php -m | findstr /I sodium
```

Для чистой установки используется `HASH_VERIFY=true`: Laravel отклоняет хэш, созданный другим алгоритмом. При обновлении с 0.13.12 скрипт может временно установить `HASH_VERIFY=false`, чтобы существующие bcrypt-хэши проверялись и после успешного входа автоматически заменялись Argon2id-хэшами. После завершения миграции верните `HASH_VERIFY=true`. PHP без поддержки Argon2id не сможет проверить уже существующий Argon2id-хэш, поэтому перенос такой базы требует PHP-сборку с Argon2id. Для bcrypt новые пароли ограничиваются 72 байтами, чтобы исключить его штатное усечение.

## База CMS и расширения PHP

Для локальной разработки и тестов можно использовать SQLite:

```env
DB_CONNECTION=sqlite
DB_DATABASE=database/database.sqlite
DB_BUSY_TIMEOUT=5000
```

`DB_BUSY_TIMEOUT=5000` позволяет кратковременно подождать освобождения файла вместо немедленной ошибки `database is locked`. WAL и `synchronous=NORMAL` автоматически не включаются. SQLite не рекомендуется для публичной production-установки KaevCMS. Требуется `pdo_sqlite`.

Для MySQL/MariaDB:

```env
DB_CONNECTION=mysql
DB_HOST=127.0.0.1
DB_PORT=3306
DB_DATABASE=kaevcms
DB_USERNAME=kaevcms
DB_PASSWORD=change_me
```

MySQL и MariaDB являются рекомендуемыми базами для публичной установки. Требуется `pdo_mysql`. При `GAME_ADAPTER=mobius` расширение `pdo_mysql` требуется независимо от драйвера основной базы CMS.

PostgreSQL рассматривается как будущая поддерживаемая база, но пока не заявлена как проверенная production-конфигурация.

## Сессии, кэш и очередь

По умолчанию один экземпляр CMS использует:

```env
SESSION_DRIVER=file
CACHE_STORE=file
QUEUE_CONNECTION=sync
```

Значение `QUEUE_CONNECTION` можно оставить `sync`: KaevCMS выбирает соединение для автоматической почты самостоятельно через настройку в административной панели.

Доступны три режима:

- синхронный — не требует дополнительных процессов;
- асинхронный — использует встроенный Laravel `background` и не требует постоянного worker;
- асинхронный с очередью в базе данных — сохраняет письма в `jobs`, поддерживает повторные попытки и требует Laravel Scheduler либо отдельный `queue:work database --queue=mail-probe,mail`.

Перед включением каждого асинхронного режима CMS ставит реальное тестовое задание и не активирует режим, если оно не было выполнено.

При запуске нескольких экземпляров приложения файловые сессии и кэш использовать нельзя. Для такой схемы нужны общее хранилище сессий и кэша, общая база CMS, единый `APP_KEY`, общее хранилище загрузок и централизованные логи. Redis является рекомендуемым вариантом, но не обязательной зависимостью односерверной установки.

## Laravel Scheduler

Мониторинг LoginServer/GameServer не требует обязательного системного расписания: публичная главная и административный дашборд автоматически обновляют устаревший снимок через CMS. Scheduler остаётся полезен, если проверки должны продолжаться при полном отсутствии посетителей.

Автоматическая очистка журналов, обработка почтовой очереди в базе данных и другие служебные задачи требуют запуска Laravel Scheduler системой каждую минуту.
KaevCMS записывает heartbeat при каждом реальном запуске Scheduler. Если отметка не обновлялась более трёх минут, административный дашборд и раздел системной информации покажут предупреждение. Для режима почты с очередью в базе дополнительно контролируются возраст ожидающих заданий и реальная активность обработчика очереди.

### Linux cron

```cron
* * * * * cd /var/www/kaevcms && php artisan schedule:run >> /dev/null 2>&1
```

### Windows Task Scheduler

Создайте задачу с повторением каждую минуту.

Программа:

```text
C:\path\to\php.exe
```

Аргументы:

```text
artisan schedule:run
```

Рабочая папка:

```text
C:\Projects\CMS\KaevCMS
```

Однократный ручной запуск `php artisan schedule:run` не создаёт постоянное расписание. Без cron или Task Scheduler мониторинг на видимых страницах продолжит работать, но очистка журналов, постоянные проверки и асинхронная почта с очередью в базе данных выполняться не будут. Синхронный и обычный асинхронный режимы почты от Scheduler не зависят.

## Резервные копии

Резервное копирование настраивается на уровне сервера или хостинга. Сохраняйте:

- базу CMS;
- `.env` и `APP_KEY`;
- `public/uploads`;
- установленные темы и модули;
- отдельно — игровые базы.

Копия должна храниться вне основного сервера. Периодически проверяйте не только создание архива, но и реальное восстановление.
