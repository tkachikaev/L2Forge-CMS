# Архитектура

```text
HTTP-запрос
  → Laravel route
  → Middleware / FormRequest
  → Controller
  → CMS database / GameServerAdapter / Mail transport
  → Blade theme
  → HTML response
```

## Разделение ответственности

- `app/` — ядро и бизнес-логика.
- `themes/` — представления публичной темы и её manifest.
- `resources/views/admin/` — независимая административная панель.
- `public/themes/` — CSS, JavaScript и изображения публичных тем.
- `public/uploads/news/` — загруженные изображения новостей со случайными именами.
- `public/uploads/pages/` — изображения многоязычных информационных страниц.
- `public/uploads/settings/` — логотип и favicon сайта со случайными именами.
- `database/` — структура собственной базы CMS.
- подключение `game` — будущая база Mobius, не используется миграциями CMS.


## Локализация

`SetApplicationLocale` выбирает язык по префиксу публичного маршрута, сессии, настройке администратора или пользователя и языку сайта по умолчанию. `LanguageManager` обнаруживает установленные пакеты в `lang/`, хранит список включённых языков и задаёт резервный язык.

```text
lang/ru/language.php
lang/ru.json
lang/en/language.php
lang/en.json
```

Публичные страницы доступны как по совместимым старым адресам, так и по явным локализованным маршрутам `/ru/...` и `/en/...`. Текст новости вынесен в `news_translations`, страницы — в `page_translations`, название игрового сервера — в `game_server_translations`. Переводимые настройки и шаблоны писем используют ключи с суффиксом локали в `cms_settings`.

Дополнительные языки устанавливаются только как проверенные файловые пакеты. Подробности: `docs/development/LOCALIZATION.md`.

## Учётные записи

В проекте предусмотрены три независимых типа учётных записей:

```text
Администратор CMS  → модель Admin, guard admin, маршруты /admin
Пользователь сайта → модель User, guard web, маршруты /login и /account
Игровой аккаунт    → будущий адаптер Login Server
```

Пользователь сайта не создаёт запись в игровой базе. Регистрация игрового аккаунта будет отдельной функцией.

Публичная авторизация включает:

- регистрацию;
- вход и выход;
- временные подписанные ссылки подтверждения email;
- повторную отправку подтверждения;
- восстановление пароля через таблицу `password_reset_tokens`;
- middleware `site.verified`, которое учитывает настройку обязательного подтверждения.

## Настройки регистрации и почты

`RegistrationSettings` читает и записывает:

```text
registration.enabled
registration.email_verification_required
```

`MailSettings` управляет SMTP-конфигурацией во время выполнения приложения. Параметры хранятся в `cms_settings`, а пароль — в зашифрованном виде. После загрузки настроек сервис обновляет Laravel mailer и очищает ранее созданный SMTP transport.

`MailTemplateSettings` объединяет стандартные шаблоны из `config/mail_templates.php` с пользовательскими переопределениями из `cms_settings`. Уведомления запрашивают готовый `MailMessage` у этого сервиса, поэтому тема, название в шапке, заголовок, текст и кнопка меняются без правки PHP-кода. Сервис проверяет список разрешённых переменных и не принимает HTML-теги.

Статус `mail.tested_at` означает, что текущий набор SMTP-параметров прошёл тестовую отправку. Любое изменение параметров сбрасывает этот статус.

Секреты приложения и баз данных остаются в `.env`. SMTP-пароль является отдельным управляемым секретом: он вводится владельцем сайта в админке и шифруется через `APP_KEY`.

## Темы

Активная тема задаётся `CMS_THEME=default`. Тема обязана содержать:

```text
themes/example/
├── theme.json
└── views/
```

Публичные файлы темы:

```text
public/themes/example/assets/
```

Тема не должна выполнять SQL-запросы и изменять данные. Она получает подготовленные данные от контроллеров и использует стандартные именованные маршруты авторизации.

## Игровые адаптеры

`GameServerAdapter` скрывает различия между сборками. Mobius, aCis и другие сборки будут отдельными классами. Так ядро и темы не зависят от названий таблиц конкретной сборки.

## Новости

Текст новости проходит через серверный allow-list sanitizer до записи в базу. Тема получает уже очищенный HTML, но модель повторно очищает его при выводе для защиты от ручного изменения базы.

Картинки-превью хранятся отдельно от изображений внутри текста:

```text
public/uploads/news/covers/
public/uploads/news/content/
```

В базе сохраняются только относительные пути и очищенный HTML.

## Страницы сайта

Модель `Page` хранит состояние публикации, расположение в навигации и порядок. Тексты, адреса и SEO-поля находятся в `page_translations`, поэтому новый язык не требует изменения схемы базы.

Публичный контроллер использует локализованный slug и резервный язык. `PageNavigation` лениво формирует коллекции `$headerPages` и `$footerPages`, не обращаясь к таблицам до выполнения миграций.

HTML страниц очищается отдельным `PageHtmlSanitizer`. Изображения разрешены только из `public/uploads/pages/content/`.

## Настройки сайта

Общие настройки хранятся в таблице `cms_settings` как пары ключ/значение. Класс `SiteSettings` управляет параметрами сайта. Игровые миры хранятся в отдельной таблице `game_servers`, а `GameServerSettings` отвечает за их нормализацию и порядок отображения.

`HomeController` объединяет каждый сохранённый игровой сервер с оперативным статусом выбранного `GameServerAdapter`. Пока подключение к отдельным базам не реализовано, один адаптер предоставляет общий демонстрационный статус. Архитектура списка уже готова для будущих отдельных подключений.

Поля подключения к игровым базам в админке пока являются отключёнными визуальными заглушками и не сохраняются.


## Журнал действий

Человекочитаемые события хранятся в таблице `audit_logs`. Ядро и модули используют единый сервис `App\Services\AuditLogger`, который:

- определяет администратора или пользователя из активного guard;
- сохраняет снимок инициатора и объекта действия;
- добавляет IP-адрес и User-Agent веб-запроса;
- рекурсивно скрывает пароли, токены, cookies и ключи;
- не прерывает основную операцию при собственной ошибке.

Журнал не заменяет `storage/logs/laravel.log`: первый предназначен для администраторов, второй — для технической диагностики разработчиками.

Подробности API: `docs/development/AUDIT_LOGGING.md`.
